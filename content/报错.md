```powershell
# Define a list of public DNS servers
$dnsServers = @(
    "114.114.115.115",
    "223.5.5.5",
    "223.6.6.6",
    "180.76.76.76",
    "8.8.8.8",
    "114.114.114.114"
)

$proxy = "http://127.0.0.1:7890"

# Set the default domain to raw.githubusercontent.com
$defaultDomain = "raw.githubusercontent.com"

# Prompt the user to enter a domain name
$domain = Read-Host "Please enter a domain name (e.g., $defaultDomain)"

# If the user does not enter anything, use the default domain
if ([string]::IsNullOrWhiteSpace($domain)) {
    $domain = $defaultDomain
}

# Function to resolve a domain using a specific DNS server
function Resolve-Domain {
    param (
        [string]$domain,
        [string]$dnsServer
    )
    try {
        # Use nslookup to resolve the domain with the specified DNS server
        $resultLines = (nslookup $domain $dnsServer 2>&1) -split "`n"
        $ipAddresses = @()
        foreach ($line in $resultLines) {
            if ($line -match "\b(?:(?:25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)\.){3}(?:25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)\b") {
                $ipAddresses += $matches[0]
            }
        }

        if ($ipAddresses.Count -ge 4) {
            $lastFourIPs = $ipAddresses[-4..-1]
        } else {
            $lastFourIPs = $ipAddresses
        }
        return $lastFourIPs
    } catch {
        Write-Host "Failed to run instruction: nslookup $domain $dnsServer"
    }
    return $null
}

# Function to test connectivity to an IP address
function Test-Connectivity {
    param (
        [string]$ipAddress
    )
    try {
        # Ping the IP address to test connectivity
        $pingResult = Test-Connection -ComputerName $ipAddress -Count 2 -ErrorAction Stop
        return $pingResult.Status -eq "Success"
    } catch {
        return $false
    }
}

# Iterate through each DNS server and resolve the domain
$resolvedIPs = @()
if (-not [string]::IsNullOrWhiteSpace($domain)) {
    foreach ($dnsServer in $dnsServers) {
        Write-Host "Resolving $domain using DNS server $dnsServer..."
        $ipAddresses = Resolve-Domain -domain $domain -dnsServer $dnsServer
        if ($ipAddresses) {
            Write-Host "Resolved IPs: $($ipAddresses -join ', ')"
            # Test connectivity to each resolved IP
            foreach ($ipAddress in $ipAddresses) {
                $curlCommand = "curl.exe -x $proxy -I -m 5 http://$ipAddress 2>&1"
                $result = & ([ScriptBlock]::Create($curlCommand))
                if ($result) {
                    Write-Host "IP $ipAddress is reachable."
                    $resolvedIPs += $ipAddress
                } else {
                    Write-Host "IP $ipAddress is not reachable."
                }
            }
        }
    }
} else {
    Write-Host "No domain entered. Exiting..."
    exit
}

# If at least one reachable IP is found, update the hosts file
if ($resolvedIPs.Count -gt 0) {
    $hostsFilePath = "$env:SystemRoot\System32\drivers\etc\hosts"
    $uniqueIPs = $resolvedIPs | Select-Object -Unique
    foreach ($ipAddress in $uniqueIPs){
    	$entry = "$ipAddress $domain"
        # Add the domain-IP mapping to the hosts file
        Add-Content -Path $hostsFilePath -Value $entry
        Write-Host "Successfully added the following entry to the hosts file:"
        Write-Host $entry
    }
} else {
    Write-Host "Failed to resolve a reachable IP for $domain."
}

# Keep the command window open after execution
Write-Host "Press any key to exit..."
$null = $Host.UI.RawUI.ReadKey("NoEcho,IncludeKeyDown")
```

# claude => 安装后启动失败

## 如何避免


1. 升级全局包时使用 "干净" 策略，Bun 全局安装出现问题时，先卸载再重装比直接升级更可靠：

```cmd
  bun remove -g @anthropic-ai/claude-code && bun add -g @anthropic-ai/claude-code
```

  升级等同于原地替换，出问题会留残骸。卸载清空目录，下次安装从零开始。

2. 注意 optionalDependencies 的下载状态

  claude-code 的 222 MB 二进制来自一个可选依赖包，下载失败时 npm/bun 默认不报错。安装后立即验证是个好习惯：

```cmd
  claude --version
```

  如果报错，说明 postinstall 没跑完，不要等到下次用才发现。

3. 安装时不要中断

  大包（二进制 ~222 MB）下载+写入需要时间，Ctrl+C 可能正好断在 rename→copy 之间，留下 .old 残骸。如果网络不稳定，宁愿等着也不要中途取消。

## 出现后如何修复

这类错误的共性模式是：postinstall 脚本在 rename 旧文件后、放置新文件前中断。排查思路：

第 1 步：定位 bin shim 报错时实际在找什么

```bash
  cat ~/.bun/install/global/node_modules/@anthropic-ai/claude-code/package.json | grep '"bin"'
```
  → 看到 "claude": "bin/claude.exe"，就去看那个文件在不在。

第 2 步：检查 bin/ 目录里有 .old.* 残骸

```bash
 ls -la ~/.bun/install/global/node_modules/@anthropic-ai/claude-code/bin/
```
 
看到 claude.exe.old.\<timestamp\> 说明上一次更新中断了。

第 3 步：找原生二进制包中有没有可用的

```bash
ls -la ~/.bun/install/global/node_modules/@anthropic-ai/claude-code-win 32-x 64/
```
如果这里也有 .old → 改名回来就行。如果这里什么都没有 → 需要重新下载。

第 4 步：修复

情况 A：.old 文件存在 → 改名回来即可（最快，无需重新下载 222 MB）

```bash
mv .../bin/claude.exe.old.* .../bin/claude.exe
```

情况 B：没有 .old 残骸 → 重新安装

```bash
bun remove -g @anthropic-ai/claude-code
bun add -g @anthropic-ai/claude-code
claude --version  # 验证
```

核心思路：CLI 工具包通常就是"一个二进制 + 一个安装脚本"，错误模式有限。先看 bin/ 目录有没有 .old 残骸，90% 的情况改名就行。

# claude code 的 hooks 在 windows 报错

## 1. Bash 钩子 → Windows 兼容性（主要问题）

**文件：** `~/.claude/settings.json`

**问题描述：**  
5 个 bash 钩子（`cbm-code-discovery-gate` + 4 个 `cbm-session-reminder` 调用）在没有正确 shell 的情况下被 PowerShell 执行。Bash heredoc 语法（`<< 'REMINDER'`）导致了以下解析错误：

```
位置 行:1 字符: 102 解析错误
```

**修复：**  
所有 bash 钩子命令现在都以 `bash` 为前缀（例如 `bash ~/.claude/hooks/cbm-session-reminder`），因此 PowerShell 通过 `bash.exe` 调用它们，而 `bash.exe` 在此系统上是可用的。

---

## 2. Hookify Python stdin 编码（修复 JSON 解析错误）

**文件：**  
`stop.py`、`pretooluse.py`、`posttooluse.py`、`userpromptsubmit.py`（hookify 插件）

**问题描述：**  
在 Windows 中文版上，Python 的 `sys.stdin` 默认使用系统 ANSI 代码页（`cp936`），但 Claude Code 通过管道传输的是 UTF-8 JSON。缺少编码匹配导致 `json.load(sys.stdin)` 抛出以下错误：

```
Expecting ',' delimiter
```

**修复：**  
在所有 4 个脚本中添加了 `sys.stdin.reconfigure(encoding='utf-8')`。

---

## 3. Hookify config_loader 路径

**文件：** `config_loader.py`

**问题描述：**  
`load_rules()` 使用了 `'.claude/'` 相对路径，如果当前工作目录（CWD）不是主目录，可能会导致路径解析失败。

**修复：**  
现在使用 `os.path.expanduser('~')` 来构建绝对路径。


# omx => winodws 环境下 --madax 启动时 bun is not installed in %PATH%

## 问题现象

在 `bun.exe` 之外的路径启动 `omx`，例如 `omx --madax --high`
出现

```cmd
[omx] madmax isolated state: C:\Users\<user>\.omx-runs\run-...
error: bun is not installed in %PATH%

Please run the following command, or double check %PATH% is right.

    powershell -c "irm bun.sh/install.ps1|iex"

[omx] codex exited with code 255
```

## 定位问题（缩小范围）

PowerShell profile 中已经做了会话级 PATH 注入，典型有效路径包括：

```text
C:\Users\<user>\Applications\bun-windows-x64-profile
C:\Users\<user>\.bun\bin
C:\Users\<user>\Applications\psmux
C:\Users\<user>\Applications\PowerShell7
```

在带 profile 的 PowerShell 7 / psmux tmux 会话中验证：

```text
bun   -> C:\Users\<user>\Applications\bun-windows-x64-profile\bun.exe
codex -> C:\Users\<user>\.bun\bin\codex.exe
omx   -> C:\Users\<user>\.bun\bin\omx.exe
tmux  -> C:\Users\<user>\Applications\psmux\tmux.exe
```

因此问题不是 profile 没加载，也不是 Bun 本体缺失。

## 复现问题

用 psmux/tmux 开持久会话后，在会话内执行：

```powershell
Set-Location 'C:\Users\<user>\Applications\PowerShell7'
Get-Command bun,codex,omx,tmux
omx --madmax --high --version
```

结果：

- `Get-Command` 能找到 `bun/codex/omx/tmux`
- `omx --madmax --high --version` 仍报 `bun is not installed in %PATH%`

这说明错误发生在 OMX 启动 Codex 子进程时，而不是用户 shell 的初始 PATH。

## 根因

OMX 的 `runCodex()` 会调用 `prependOmxRuntimeCommandShimToEnv()`，给 Codex 子进程 PATH 前面加一个 runtime shim：

```text
C:\Users\<user>\.omx-runs\run-...\.omx\runtime\bin
```

旧逻辑只读取：

```js
const currentPath = typeof env.PATH === "string" ? env.PATH : "";
```

但 Windows 环境变量实际键名经常是：

```js
env.Path
```

在 Node.js 中，`process.env.PATH` 可能通过大小写不敏感访问返回值；但一旦 OMX 将 `process.env` 展开成普通对象：

```js
{ ...process.env }
```

这个普通对象只保留真实键 `Path`，不再有 `PATH` 别名。旧代码读取 `env.PATH` 失败，认为当前 PATH 为空，最终传给 Codex 的 PATH 只剩：

```text
C:\Users\<user>\.omx-runs\run-...\.omx\runtime\bin
```

`codex.exe` 是 Bun 生成的 Windows shim；当它在这个被截断的 PATH 中启动时找不到 `bun.exe`，于是报：

```text
bun is not installed in %PATH%
```

## 诊断证据

通过临时放置 `codex.cmd` wrapper 到 PATH 最前面，记录 OMX 实际传给 Codex 子进程的 PATH，捕获到：

```
PATH=C:\Users\<user>\.omx-runs\run-...\.omx\runtime\bin
ARGS="--version" "--dangerously-bypass-approvals-and-sandbox" "-c" "model_reasoning_effort=""high""" ...
REAL_EXIT=255
```

这证明 Codex 子进程 PATH 被 OMX 截断。

## 修复方案

补丁文件：`C:\Users\<user>\.bun\install\global\node_modules\oh-my-codex\dist\cli\index.js`
备份文件：`C:\Users\<user>\.bun\install\global\node_modules\oh-my-codex\dist\cli\index.js.bak-pathfix-20260613182139`

修复前：

```javascript
export function prependOmxRuntimeCommandShimToEnv(cwd, env, omxBin, nodePath = process.execPath) {
    const shimDir = ensureOmxRuntimeCommandShim(cwd, omxBin, nodePath);
    const currentPath = typeof env.PATH === "string" ? env.PATH : "";
    return {
        ...env,
        PATH: currentPath ? `${shimDir}${delimiter}${currentPath}` : shimDir,
        OMX_ENTRY_PATH: omxBin,
        OMX_STARTUP_CWD: typeof env.OMX_STARTUP_CWD === "string" && env.OMX_STARTUP_CWD.trim()
            ? env.OMX_STARTUP_CWD
            : cwd,
    };
}
```

修复后：

```javascript
export function prependOmxRuntimeCommandShimToEnv(cwd, env, omxBin, nodePath = process.execPath) {
    const shimDir = ensureOmxRuntimeCommandShim(cwd, omxBin, nodePath);
    const currentPath = typeof env.PATH === "string"
        ? env.PATH
        : typeof env.Path === "string"
            ? env.Path
            : typeof env.path === "string"
                ? env.path
                : "";
    const nextPath = currentPath ? `${shimDir}${delimiter}${currentPath}` : shimDir;
    return {
        ...env,
        PATH: nextPath,
        ...(process.platform === "win32" ? { Path: nextPath } : {}),
        OMX_ENTRY_PATH: omxBin,
        OMX_STARTUP_CWD: typeof env.OMX_STARTUP_CWD === "string" && env.OMX_STARTUP_CWD.trim()
            ? env.OMX_STARTUP_CWD
            : cwd,
    };
}
```

# omx => windows 上出现  hook exited with code 1

## 问题现象

在 windows + psmux/tmux 中，启动 `omx` 后，输入 `你好`，codex tui 报：

```
⚠ MCP startup interrupted. The following servers were not initialized: codex_apps

• SessionStart hook (failed)
  error: hook exited with code 1

• UserPromptSubmit hook (failed)
  error: hook exited with code 1

• Stop hook (failed)
  error: hook exited with code 1
```

## 定位问题（缩小范围）

Windows 11 环境确认：

```
PowerShell 7: C:\Users\<user>\Applications\PowerShell7\pwsh.exe
psmux/tmux: C:\Users\<user>\Applications\psmux\tmux.exe
Bun: C:\Users\<user>\Applications\bun-windows-x64-profile\bun.exe
Codex/OMX: C:\Users\<user>\.bun\bin\codex.exe / omx.exe
OMX 源码位置: C:\Users\<user>\.bun\install\global\node_modules\oh-my-codex\
工作目录: C:\Users\<user>\Applications\PowerShell7
```

排除项：

1. **不是 Bun 未安装**：`bun --version` 正常工作
2. **不是用户 profile 未加载**：`Get-Command` 能找到所有可执行文件
3. **不是 PATH 配置错误**：用户 PATH 包含 Bun 目录且优先级正确
4. **不是 psmux/tmux 环境问题**：普通 PowerShell 窗口同样复现

初步判断：

问题定位在 **OMX 启动 Codex 子进程时的环境变量传递**，而非用户 shell 初始化。

## 复现问题

在 psmux/tmux 持久会话中：

```powershell
# 1. 切换到工作目录
Set-Location 'C:\Users\<user>\Applications\PowerShell7'

# 2. 验证用户环境正常
Get-Command bun, codex, omx, tmux
# 输出：全部找到，路径正确

# 3. 验证 Bun 本身可用
bun --version
# 输出：1.2.15

# 4. 触发问题
omx --madmax --high --version
```

出现结果

```text
[omx] madmax isolated state: C:\Users\y86133\.omx-runs\run-20260613102227-f283
error: bun is not installed in %PATH%
[omx] codex exited with code 255
```

对比测试

```powershell
# 不带 --madmax 正常
omx --version
# 输出：codex-cli 0.133.0

# 带 --madmax 失败
omx --madmax --high --version
# 输出：error: bun is not installed in %PATH%
```

**关键发现**：`--madmax` 参数触发了 OMX 的隔离环境机制，导致问题暴露。

## 根因挖掘

1. 捕获子进程实际环境变量

创建 wrapper 脚本 `C:\Users\y86133\codex-debug.cmd`：

```batch
@echo off
echo [DEBUG] ========== Codex Wrapper Started ========== >> C:\temp\codex-debug.log
echo [DEBUG] TIME: %DATE% %TIME% >> C:\temp\codex-debug.log
echo [DEBUG] PATH=%PATH% >> C:\temp\codex-debug.log
echo [DEBUG] ARGS=%* >> C:\temp\codex-debug.log
echo [DEBUG] Current directory: %CD% >> C:\temp\codex-debug.log

"C:\Users\<user>\.bun\bin\codex.exe" %*

echo [DEBUG] EXIT CODE: %ERRORLEVEL% >> C:\temp\codex-debug.log
echo [DEBUG] ========== Codex Wrapper Ended ========== >> C:\temp\codex-debug.log
exit %ERRORLEVEL%
```

临时将 wrapper 所在目录插入 PATH 最前面，重新运行 `omx --madmax --high --version`。

2. 日志分析

查看 `C:\temp\codex-debug.log`：

```text
[DEBUG] ========== Codex Wrapper Started ==========
[DEBUG] TIME: 2026-06-13 10:22:27
[DEBUG] PATH=C:\Users\y86133\.omx-runs\run-20260613102227-f283\.omx\runtime\bin
[DEBUG] ARGS="--version" "--dangerously-bypass-approvals-and-sandbox" "-c" "model_reasoning_effort=high"
[DEBUG] Current directory: C:\Users\y86133\Applications\PowerShell7
[DEBUG] EXIT CODE: 255
[DEBUG] ========== Codex Wrapper Ended ==========
```

**关键发现**：

- PATH 只剩下 `.omx\runtime\bin`
- 原有的用户 PATH（包含 Bun 目录）完全丢失

```javascript
    const shimDir = ensureOmxRuntimeCommandShim(cwd, omxBin, nodePath);
    
    // Windows: 兼容 PATH / Path / path 大小写变体
    const currentPath = typeof env.PATH === "string"
        ? env.PATH
        : typeof env.Path === "string"
            ? env.Path
            : typeof env.path === "string"
                ? env.path
                : "";
    
    const nextPath = currentPath ? `${shimDir}${delimiter}${currentPath}` : shimDir;
    
    return {
        ...env,
        PATH: nextPath,
        // Windows: 同时设置规范大写 Path 键，确保兼容性

        OMX_ENTRY_PATH: omxBin,

            ? env.OMX_STARTUP_CWD
            : cwd,
    };
}
```

## 根因挖掘

### 时间线冲突

PowerShell 进程启动后，**事件顺序**是：

1. 进程加载，CWD 继承自父进程（`explorer.exe` 的 `C:\Windows\System32`）
2. PowerShell 加载 profile 脚本（`$PROFILE`）
3. 执行 `-command "Set-Location -literalPath '...'"`

**第二步 profile 运行时，第三步的 `Set-Location` 还没执行**。此时 `(Get-Location).Path` 返回的仍然是进程默认的 `C:\Windows\System32`。

没有 profile → 不进入 `if (Test-Path $pwshExe)` 块 → 不启动 pwsh → `-command "Set-Location ..."` 正常执行 → 用户看到的就是目标目录。

## 修复方案

不依赖 `(Get-Location).Path`，直接解析命令行参数：

```powershell
$cmdArgs = [Environment]::GetCommandLineArgs()
$currentDir = $null

# 检查 1：-WorkingDirectory（Windows Terminal、VS Code 终端、pwsh 自调用）
$wdIdx = [array]::IndexOf($cmdArgs, '-WorkingDirectory')
if ($wdIdx -ge 0 -and $wdIdx -lt $cmdArgs.Length - 1) {
    $currentDir = $cmdArgs[$wdIdx + 1]
}

# 检查 2：-command "Set-Location -literalPath '<path>'"（Windows 右键菜单）
if (-not $currentDir) {
    $cmdIdx = [array]::IndexOf($cmdArgs, '-command')
    if ($cmdIdx -ge 0 -and $cmdIdx -lt $cmdArgs.Length - 2) {
        $slIdx = [array]::IndexOf($cmdArgs, 'Set-Location', $cmdIdx + 1)
        if ($slIdx -ge 0) {
            $lpIdx = [array]::IndexOf($cmdArgs, '-literalPath', $slIdx + 1)
            if ($lpIdx -ge 0 -and $lpIdx -lt $cmdArgs.Length - 1) {
                $currentDir = $cmdArgs[$lpIdx + 1] -replace "^['\""]|['\""]$", ''
            }
        }
    }
}

# 回退：用户主目录
if (-not $currentDir) {
    $currentDir = $HOME
}
```

## 注意事项

1. **`-WorkingDirectory` 是 pwsh 独有参数**，PowerShell 5.1 不认识它。所以右键菜单使用 `-command Set-Location` 而非 `-WorkingDirectory`。两种机制都要处理。
2. **引号剥离**：`GetCommandLineArgs()` 保留参数中的单引号/双引号。提取路径后要用正则 `^['"]|['"]$` 去掉。
3. **PowerShell 参数解析模式**：方法调用如 `[System.IO.Directory]::GetCurrentDirectory()` 作为函数参数传递时必须用 `()` 包裹，否则触发生成器错误 `"(" 后面应为表达式`。
4. **场景覆盖**：非右键启动时命令行可能没有 `-command` 参数，回退到 `$HOME` 比回退到 `Get-Location`（可能为 System32）更安全。
5. **进程 CWD vs PSDrive 位置**：`Get-Location` 返回 PowerShell 提供程序位置，`[Environment]::CurrentDirectory` 返回 .NET 进程当前目录。在 profile 加载时刻两者都可能是 System32。
6. **`-Command` 参数组装差异**：不同 Windows 版本的右键菜单注册表项可能参数顺序略有不同（`-noexit -command "..."` vs `-Command "..."` vs `-command Set-Location -literalPath...`），解析时需要考虑。

## 根因挖掘

### 根因一（CodeGraph 适用）：`.cmd` 包装器导致管道断裂

OMP 用 Bun.spawn 启动 `.cmd` 文件时，Windows 上的 stdin/stdout 管道未正确传递给子进程。CodeGraph 的 `codegraph` 命令解析 PATH 到 `codegraph.cmd`，内嵌的 `node.exe` 收不到 OMP 的 MCP 消息，进程启动后立即退出 → `Transport closed`。

**修复**：跳过 `.cmd`，直接捆绑的 `node.exe`：

```json
// ~/.omp/agent/mcp.json
"codegraph": {
  "command": "C:\\Users\\y86133\\Applications\\codegraph-win32-x64\\node.exe",
  "args": [
    "--liftoff-only",
    "C:\\Users\\y86133\\Applications\\codegraph-win32-x64\\lib\\dist\\bin\\codegraph.js",
    "serve", "--mcp"
  ]
}
```

`--liftoff-only` 是 codegraph 自定义 Node.js 构建的参数，指示其加载指定 JS 文件作为入口。

### 根因二（codebase-memory-mcp 适用）：243 MB 大型二进制冷启动超 30 s 默认 MCP 超时

二进制直接测试（shell pipe / Bun.spawn / child_process.spawn）全部 <1 s 响应。但在 OMP 环境下，进程冷启动（243 MB PE 镜像加载 + 反病毒扫描 + 运行时初始化 + 数据库初始化）消耗时间超过 OMP 的 `DEFAULT_MCP_TIMEOUT`（30 s）。

OMP 的 MCP 客户端在发送 `initialize` 后等待响应，若超过 30 s 未收到，判定为超时。此时二进制实际上可能仍在启动中，尚未从 stdin 读取到 initialize 消息。

**修复**：通过 `"timeout": 0` 禁用 OMP 客户端侧 MCP 超时：

```json
"codebase-memory-mcp": {
  "command": "C:\\Users\\y86133\\Applications\\codebase-memory-mcp\\codebase-memory-mcp.exe",
  "timeout": 0
}
```

该设置告诉 OMP 无限等待 MCP 服务器响应。实际上冷启动完成后，二进制立即响应（后续调用均正常）。

**关键证据链**：

1. Shell pipe 测试 0.6 s 响应——但这是热启动（管道已包含数据）
2. OMP 启动时，进程先 spawn，然后 OMP 发送 initialize——期间有时间差
3. 243 MB 文件加载 + 反病毒扫描可能在 OMP 环境下显著延长
4. `"timeout": 0` 是唯一生效的修复

### 辅助发现：僵尸进程与 SQLite WAL 锁

此前版本的 `launcher.cmd` 设计了清理逻辑：

```cmd
taskkill /F /IM "codebase-memory-mcp.exe" >nul 2>&1
del /f /q %USERPROFILE%\.cache\codebase-memory-mcp\*.db-shm 2>nul
del /f /q %USERPROFILE%\.cache\codebase-memory-mcp\*.db-wal 2>nul
```

每次失败连接会留下残留进程，持有 SQLite 数据库的 WAL 锁（`.db-shm` / `.db-wal` 文件）。新实例启动时卡在等锁释放 → 无响应 → OMP 超时。

**但这不是当前修复的根因**——清理后即使无 WAL 锁竞争，默认超时仍然不足。

## 注意事项

### Windows .cmd 文件与 MCP

OMP 在 Windows 上用 `Bun.spawn` 启动 MCP 服务器。`.cmd` / `.bat` 文件作为命令时，OMP 会添加 `cmd.exe /d /s /c` 前缀，但此包装链的 stdin/stdout 管道可能不正常。

**经验**：MCP 服务器的 `command` 应始终指向原生可执行文件（`.exe`）或 Node.js 脚本（`node script.js`），而非 `.cmd` 包装器。

### OMP MCP timeout 参数

`"timeout": 0` 禁用 OMP 客户端侧 MCP 超时。默认值 30000 ms 对大多数 MCP 服务器足够，但对于：

- 大型二进制（>100 MB）冷启动
- .NET 自包含应用（CLR 初始化 JIT）
- 初次数据库初始化或建索引
- 反病毒软件扫描延时

需设为 0 或一个更大的毫秒值（如 `120000` 表示 2 分钟）。

也可以全局设置环境变量：

```bash
set OMP_MCP_TIMEOUT_MS=120000
```

```bash

del /f /q %USERPROFILE%\.cache\codebase-memory-mcp\*.db-shm 2>nul
```
