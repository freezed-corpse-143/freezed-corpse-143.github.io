# nvidia 显卡驱动安装

安装完验证显卡驱动安装成功，命令行输入`nvidia-smi`
## windows  环境

1. 查看本机显卡版本型号，`win`+`R`，输入`DxDiag`，在`显示`或`呈现`栏包含显卡信息
2. 查看显卡版本与显卡驱动版本的兼容性，在 https://www.nvidia.cn/geforce/drivers/ ，查看本机显卡支持的显卡驱动，根据搜索结果下载最新的`NVIDIA Studio`，直接安装（不用管过去安装的驱动），重启。

# cuda 安装

前提： [[环境配置#nvidia 显卡驱动安装]]，[[环境配置#aria2 安装]]
安装完测试，命令行输入
```bash
nvcc --version
```
## windows 环境

查看显卡驱动与Cuda的兼容性，命令行输入`nvidia-smi`，右上角显示显卡驱动程序先支持的最高版本。
在 https://developer.nvidia.com/cuda-toolkit-archive ，确定版本，在官方页面，选择系统，然后下载`exe`文件
如果已经安装，就卸载旧版本`cuda toolkit`，控制面板就可以。

## linux 环境
在 https://developer.nvidia.com/cuda-toolkit-archive ，确定版本，在官方页面，选择系统，然后下载`run`文件。

举例说明
```bash
apt install nvidia-modprobe -y

aria2c -x 16 -s 16 -c "https://developer.download.nvidia.com/compute/cuda/12.8.0/local_installers/cuda_12.8.0_570.86.10_linux.run"

sh cuda_12.8.0_570.86.10_linux.run
```

`Abort/Continue`选择`Continue`，不要勾选自带的`driver`

配置环境变量
```ini
export PATH=/usr/local/cuda/bin:$PATH
export LD_LIBRARY_PATH=/usr/local/cuda/lib64:$LD_LIBRARY_PATH
```

### 卸载

1. 提取卸载程序

   ```bash
   bash **.run --extract=ANY_ABSOLUTE_PATH
   ```

2. 在`ANY_ABSOLUTE_PATH/bin`，有个程序`cuda-uninstaller`

   ```bash
   chmod +x ANY_ABSOLUTE_PATH/bin/cuda-uninstaller
   
   ANY_ABSOLUTE_PATH/bin/cuda-uninstaller
   ```

# torch 安装

前提： [[#nvidia 显卡驱动安装]]，

torch wheel 自带运行时，除非你：

- 自己编译 CUDA 项目
- 编译 xformers
- 编译 flash-attn
- 编译 custom op

否则根本不看 nvcc。

1. 确定 `cuda` 版本，
2. 运行命令

```bash
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu121
```

安装的是：

> PyTorch 官方自带 CUDA 12.1 runtime 的 wheel

它会自己带：

- cudart
- cuBLAS
- cuDNN
- NCCL
- 等运行库

所以：

|项目|是否依赖本机 CUDA Toolkit|
|---|---|
|运行 PyTorch|基本不依赖|
|训练/推理|不依赖|
|ComfyUI|不依赖|
|Stable Diffusion|不依赖|
|transformers|不依赖|
|编译 CUDA 扩展|才依赖 nvcc|

真正重要的是：

你 NVIDIA 驱动够不够新

看` nvidia-smi`

# elasticsearch 安装

## linux 安装

1. 安装Java：如果已经安装则跳过使用java -version查看

```bash
sudo apt update
sudo apt install openjdk-8-jdk -y
java -version
```

2. 添加**添加Elasticsearch存储库**：由于Elasticsearch软件包不包含在Ubuntu 22.04的默认存储库中，需要添加官方存储库

   3. **安装依赖**：

```bash
sudo apt install curl wget gnupg2 -y
```

   4. **添加Elasticsearch GPG密钥**：

```bash
curl -fsSL https://artifacts.elastic.co/GPG-KEY-elasticsearch | sudo gpg --dearmor -o /usr/share/keyrings/elastic.gpg
```

   5. **添加Elasticsearch存储库**：选择一个与jdk匹配的版本

```bash
echo "deb [signed-by=/usr/share/keyrings/elastic.gpg] https://artifacts.elastic.co/packages/7.x/apt stable main" | sudo tee -a /etc/apt/sources.list.d/elastic-7.x.list
```

   6. **更新APT源**：

```bash
sudo apt update
```

7. **安装Elasticsearch**：

```bash
sudo apt install elasticsearch -y
```

8. 启动服务

```bash
sudo sysctl start elasticsearch

curl -X GET 'http://localhost:9200'
```
## windows 安装

>1. Elasticsearch 7.x 及之前版本支持 Java 8 。
>2. Elasticsearch 7.17.0 版本开始，推荐使用 Elasticsearch Java API，并且这个版本弃用了 High Level REST Client，转而使用新的客户端 ElasticsearchClient 。

1. 进入版本选择网站 https://www.elastic.co/cn/downloads/past-releases#elasticsearch

2. 选择版本：

3. 解压到本地文件夹中

4. 配置其config/elasticsearch.yml，修改前备份

```ini
path.data: #换成自己的路径
#设置日志文件的存储路径
path.logs: #换成自己的路径
```

5. 现在默认使用自带的jdk，如果环境变量中有`JAVA_HOME`，请在`bin\elasticsearch-env.bat`中，注销掉相关分支程序

6. 进入bin目录

```cmd
elasticsearch.bat
```

## SSH连接容器

### 使用Dockerfile文件配置镜像

1. 创建Dockerfile文件。

```dockerfile
# Dockerfile
FROM nvidia/cuda:12.8.0-cudnn-runtime-ubuntu24.04

# 设置环境变量
ENV DEBIAN_FRONTEND=noninteractive

# 更新包列表并安装必要软件
RUN apt-get update && apt-get install -y \
    openssh-server \
    sudo \
    curl \
    wget \
    vim \
    nano \
    git \
    build-essential \
    python3 \
    python3-pip \
    net-tools \
    iputils-ping \
    && rm -rf /var/lib/apt/lists/*

# 配置SSH
RUN mkdir /var/run/sshd

# 设置root密码（你可以修改这个密码）
RUN echo 'root:123456' | chpasswd

# 配置SSH允许root登录
RUN sed -i 's/#PermitRootLogin prohibit-password/PermitRootLogin yes/' /etc/ssh/sshd_config

# 修复SSH PAM问题
RUN sed -i 's@session\s*required\s*pam_loginuid.so@session optional pam_loginuid.so@g' /etc/pam.d/sshd

# 创建普通用户（可选但推荐）
RUN useradd -m -s /bin/bash test && \
    echo 'test:123456' | chpasswd && \
    usermod -aG sudo test

# 设置工作目录
WORKDIR /workspace
```

### 手动配置

1. 使用`docker-compose up -d`启动一个具有端口映射`2222:22`的容器。

2. 使用`docker exec -it ssh_container bash`进入容器命令行，运行

```bash
apt-get update && apt-get install -y wget curl tar gzip ca-certificates procps openssh-server && \
mkdir -p /run/sshd && chmod 0755 /run/sshd && \
echo 'root:123456' | chpasswd && \
ssh-keygen -A && \
cat > /etc/ssh/sshd_config << 'EOF'
Port 2222
AddressFamily any
ListenAddress 0.0.0.0
ListenAddress ::
PermitRootLogin yes
PasswordAuthentication yes
UsePAM yes
PidFile /run/sshd.pid
Subsystem sftp /usr/lib/openssh/sftp-server
EOF
rm -rf /root/.ssh && \
mkdir -p /root/.ssh && \
cp /tmp/ssh/* /root/.ssh/ 2>/dev/null || true && \
chown -R root:root /root/.ssh && \
chmod 700 /root/.ssh && \
chmod 600 /root/.ssh/* 2>/dev/null || true && \
/usr/sbin/sshd -f /etc/ssh/sshd_config
```

```bash
# MODEL_PATH=gpt2

# 方案2: 使用 TinyLlama (1.1B，但比 Qwen3B 小很多)
# MODEL_PATH=TinyLlama/TinyLlama-1.1B-Chat-v1.0

# 方案3: 使用阿里巴巴的极简模型（如果有）
```

5. 运行脚本

```bash
bash run_qwen2.5_7b_grpo_4gpu.sh
```

   

这个脚本是使用 **GRPO (Group Relative Policy Optimization)** 算法进行强化学习训练。以下是主要参数的含义分类说明：

### 🎯 算法核心参数

| 参数                         | 值      | 含义                                                         |
| ---------------------------- | ------- | ------------------------------------------------------------ |
| `algorithm.adv_estimator`    | `grpo`  | 使用 GRPO 算法（Group Relative Policy Optimization），是 PPO 的变体，不使用价值网络 |
| `algorithm.use_kl_in_reward` | `False` | 不将 KL 散度直接加入奖励，而是通过单独的 KL loss 约束策略更新 |

### 🎭 Actor（策略模型）参数

| 参数                                 | 值           | 含义                                              |
| ------------------------------------ | ------------ | ------------------------------------------------- |
| `actor.optim.lr`                     | `3e-5`       | 学习率                                            |
| `actor.ppo_mini_batch_size`          | `16`         | PPO 每次更新使用的样本数（4 GPU × 4 micro-batch） |
| `actor.ppo_micro_batch_size_per_gpu` | `2`          | 每个 GPU 的微批次大小，控制显存占用               |
| `actor.use_kl_loss`                  | `True`       | 使用 KL 散度损失约束策略更新                      |
| `actor.kl_loss_coef`                 | `0.001`      | KL 损失系数，控制约束强度                         |
| `actor.kl_loss_type`                 | `low_var_kl` | KL 损失计算方式，低方差版本                       |

### 🔄 Rollout（采样）参数

| 参数                                        | 值     | 含义                                                         |
| ------------------------------------------- | ------ | ------------------------------------------------------------ |
| `rollout.n`                                 | `5`    | **GRPO 核心参数**：每个 prompt 采样 5 个回答，在组内计算相对奖励 |
| `rollout.tensor_model_parallel_size`        | `2`    | vLLM 的 tensor parallelism 大小                              |
| `rollout.name`                              | `vllm` | 使用 vLLM 作为推理引擎加速采样                               |
| `rollout.gpu_memory_utilization`            | `0.25` | vLLM 使用 25% 显存，为训练预留空间                           |
| `rollout.max_model_len`                     | `1536` | 最大序列长度（prompt + response）                            |
| `rollout.log_prob_micro_batch_size_per_gpu` | `2`    | 计算 log probability 时的微批次大小                          |

### 📊 数据参数

| 参数                           | 值     | 含义                                                         |
| ------------------------------ | ------ | ------------------------------------------------------------ |
| `data.train_batch_size`        | `64`   | 每个训练步的总样本数（64 个 prompt × 5 个 rollout = 320 个回答） |
| `data.max_prompt_length`       | `512`  | prompt 最大长度                                              |
| `data.max_response_length`     | `1024` | 回答最大长度                                                 |
| `data.filter_overlong_prompts` | `True` | 过滤超长 prompt                                              |

### 💾 模型与显存优化

| 参数                                  | 值           | 含义                     |
| ------------------------------------- | ------------ | ------------------------ |
| `model.lora_rank`                     | `32`         | LoRA 秩，低秩适配维度    |
| `model.lora_alpha`                    | `32`         | LoRA 缩放系数            |
| `model.target_modules`                | `all-linear` | LoRA 应用的所有线性层    |
| `model.enable_gradient_checkpointing` | `True`       | 梯度检查点，以计算换显存 |
| `actor.fsdp_config.param_offload`     | `True`       | FSDP 参数卸载到 CPU      |
| `actor.fsdp_config.optimizer_offload` | `True`       | 优化器状态卸载到 CPU     |
| `model.use_remove_padding`            | `True`       | 移除 padding 提升效率    |

### 🏃 训练控制参数

| 参数                    | 值   | 含义                                   |
| ----------------------- | ---- | -------------------------------------- |
| `trainer.total_epochs`  | `3`  | 总训练轮数                             |
| `trainer.save_freq`     | `20` | 每 20 步保存一次模型                   |
| `trainer.test_freq`     | `5`  | 每 5 步进行一次验证                    |
| `trainer.critic_warmup` | `0`  | Critic warmup 步数（GRPO 无需 critic） |

### 🔑 GRPO 核心机制

**GRPO 与 PPO 的关键区别：**

- 不需要 Critic 网络（价值函数）
- 使用 `rollout.n=5` 对每个 prompt 采样多个回答
- 在组内（同一 prompt 的 5 个回答）计算相对优势，减去组内均值并除以标准差
- 节省了价值网络的显存，适合 4×24G 配置

# devcontainer 管理容器

配置 `devcontainer.json`

```json
{
    "name": "dev",
    "image": "ubuntu:22.04",
    "customizations": {
        "zed": {
            "settings": {},
            "extensions": []
        }
    }
}
```

会自动将 `devconainer.json` 所在目录下的内容映射到容器中

这背后的原理是**数据卷（Mount）** 机制。你在 `devcontainer.json` 中配置的目录映射，本质上是在容器和宿主机（也就是你的WSL目录）之间建立了一个**“桥梁”**。

- **关系是“映射”，不是“移动”**：这个配置相当于告诉Docker：“请让容器能看到并使用WSL里的这个文件夹”。它并没有把文件夹“移动”或“复制”到容器里。
- **容器删除，桥梁拆除**：当你删除容器时，相当于拆掉了这座“桥梁”。容器这个“房子”被拆掉了，但它所连接的那条“路”（WSL中的目录）依然在原地，不受影响。
- **数据持久化**：这正是使用“绑定挂载”这种方式的优势所在——你的源代码和数据可以独立于容器的生命周期而持久保存。

所以，你可以放心地删除、重建或更新你的开发容器，只要挂载配置指向的是同一个WSL目录，容器就能再次连接到这些文件。

所以，使用 devcontainer 管理容器，而不是手动创建容器，并在容器中管理数据，一旦容器被误删除，会导致数据丢失。

同时在`devcontainer.json`中禁用配置

```json
"runArgs": [ "--name=dev" ]
```

硬编码容器名，这可能会导致后续容器命名冲突、容器创建失败等情况。正确的方法是让zed或docker自行管理容器命名。

## 声明式配置

在 NixOS 的世界里，“响应式”并不是指它会像你的手机屏幕那样自动旋转，而是指一种**声明式（Declarative）**与**确定性（Deterministic）**的设计哲学。

1. 为什么说是“声明式”配置？

在传统的 Linux 发行版（如 Ubuntu 或 Arch）中，系统配置是**过程式**的：你手动安装包、手动修改 `/etc` 下的文件。如果你搞砸了某一步，很难完全恢复到之前的状态。

NixOS 采用了类似 React 或 Flutter 的 UI 开发思路：

- **状态说明书：** 你的 `.nix` 文件描述了系统的“最终目标状态”（例如：安装了哪些软件、开启了哪些服务）。
- **原子更新：** 当你执行切换命令时，Nix 会对比当前状态与目标状态，只生成差异部分，并创建一个新的系统生成（Generation）。

------

2. 为什么修改文件后不会“自动”生效？

虽然技术上可以实现自动监控（类似 Web 开发中的热重载），但 NixOS 默认不这么做是有核心原因的：

- **安全性与稳定性：** 编译系统环境是一个相对沉重的过程。如果你正在修改配置，还没写完代码就触发自动构建，可能会导致语法错误甚至系统配置半途而废。
- **显式确认：** NixOS 的精髓在于**版本控制**。手动执行 `rebuild` 意味着你确认当前逻辑闭环，并愿意为此创建一个新的回滚点。
- **计算开销：** 评估 Nix 表达式需要消耗 CPU 和内存。频繁的后台自动构建会严重干扰你的正常工作。

------

3. 核心工作流程：从文件到系统

当你修改完 `configuration.nix` 后，标准的流程如下：

| **步骤**    | **操作**               | **发生的事**                                                 |
| ----------- | ---------------------- | ------------------------------------------------------------ |
| **1. 编写** | 修改 `.nix` 文件       | 只是在磁盘上保存了文本，系统环境无变化。                     |
| **2. 构建** | `nixos-rebuild build`  | Nix 检查语法，下载依赖，在 `/nix/store` 生成新的系统闭包。   |
| **3. 切换** | `nixos-rebuild switch` | **关键点：** 修改符号链接，将系统引导指向新版本，并实时重启相关服务。 |

------

4. 所谓的“响应式”体验体现在哪？

虽然触发动作是手动的，但**结果**是响应式的：

1. **服务自动编排：** 如果你在配置里加了一行 `services.nginx.enable = true;`，执行 switch 后，NixOS 会自动生成配置文件、创建用户、打开防火墙端口并启动服务。你不需要手动处理这些琐事。
2. **环境一致性：** 只要配置文件不变，在任何机器上重构出的系统环境几乎完全一致。
3. **瞬间回滚：** 如果新配置导致系统崩溃，你可以在启动菜单选择旧的 Generation，瞬间回到修改前的状态。

### 为什么选择 jj？

**告别 Git 的痛点：**

- 需要同时开发多个功能？创建多个工作区并行进行
- 担心操作失误？jj 记录所有操作，任何时候都能回滚

# llama.cpp

**llama.cpp** 是一个用纯 C/C++ 编写的高性能开源项目，专门用于在本地硬件上高效运行大型语言模型（LLM）。它的主要目标是让用户能够在各种硬件上以最小设置运行 AI 模型，同时提供最先进的性能。

## 主要功能特性

**技术核心**：

- 纯 C/C++ 实现，无外部依赖
- GGUF 模型格式 - 高效便携的单一文件格式
- 先进的量化技术（1.5-bit 到 8-bit）
- 跨平台硬件加速支持

**硬件支持**：

- Apple Silicon（Metal 框架优化）
- NVIDIA GPU（CUDA）
- AMD GPU（HIP）
- Intel（SYCL、OpenVINO）
- 各种 CPU 架构（x86、ARM、RISC-V）

## 解决的问题

1. **隐私保护** - 数据完全本地处理，不上传云端
2. **成本控制** - 避免昂贵的云端 API 调用费用
3. **离线使用** - 无网络环境也能运行 AI 模型
4. **硬件兼容性** - 在消费级硬件上运行原本需要高端 GPU 的模型
5. **部署灵活性** - 从嵌入式设备到服务器集群都能部署

**定位**：llama.cpp 专注于推理优化，而非训练

```bash
../configure \
    --cc=gcc \
    --prefix=/ffmpeg/dist-musl \
    --disable-autodetect \
    --disable-shared \
    --enable-static \
    --disable-ffplay \
    --disable-doc \

echo

file /ffmpeg/dist-musl/bin/ffmpeg
file /ffmpeg/dist-musl/bin/ffprobe

echo

ldd /ffmpeg/dist-musl/bin/ffmpeg || true
```

## 原理

```
You type "claude"
       │
       ▼
  Shell function (injected in .bashrc/.zshrc)
       │
       ├─ Already in tmux? ──▶ Start background monitor
       │                        Launch claude with full TUI
       │
       └─ Not in tmux? ──▶ Create tmux session transparently
                             Launch claude + monitor inside
                             Attach (looks the same to you)

  MONITOR (background, ~0% CPU):
       │
       ├─ Polls tmux pane every 5 seconds
       ├─ Detects rate limit text
       ├─ Parses reset time from message
       ├─ Waits until reset + safety margin
       ├─ Verifies Claude is still the foreground process
       └─ Sends "continue" via tmux send-keys
```
